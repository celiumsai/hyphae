## Current architecture

### Durable authority

`hyphae-storage` currently persists only KV logical mutations:

- mutation tags encode `Put` and `Delete`;
- log frames declare disk format `1`;
- the logical snapshot contains KV entries and idempotency receipts;
- backup packages the logical snapshot and rebuilds derived state on restore;
- Redb tables are rebuildable indexes, not durable authority;
- the immutable compatibility fixture proves a format-1 directory can open
  without its derived index.

This means durable vectors cannot be implemented only as a new Redb table.
They must participate in logical mutations, snapshots, recovery, compaction,
backup, restore, and compatibility behavior.

### Retrieval

`hyphae-retrieval` currently defines exact provider-neutral cosine reference
semantics over host-supplied in-memory vectors:

- floating-point scoring is `f64`;
- ordering is score descending, then binary key ascending;
- invalid input and budget exhaustion produce explicit abstention;
- the engine helper receives ephemeral `VectorRecord` slices;
- vectors are not persisted and retrieval is not exposed through HTTP v1.

### Proofs

`result-proof-v1` currently proves only KV `get` and structured `query`
results. Its codec has a versioned magic value and closed operation tags.
Retrieval proofs do not exist.

The safest initial direction is a separate `retrieval-proof-v1` contract.
ADR-0016 must accept or reject that direction before implementation.

### Public contracts

The canonical public sources are:

- `contracts/openapi/hyphae-v1.openapi.json`
- `contracts/json-schema/*-v1.schema.json`

`hyphae-contracts` packages a byte-identical mirror under
`crates/hyphae-contracts/assets/`. Tests enforce equality. TypeScript and
Python models are generated from the contract schemas. Any retrieval surface
must update the canonical files, packaged mirror, generated models, and common
conformance in one coherent change.

### Dependency boundary

`hyphae-storage` does not depend on `hyphae-retrieval`. Adding that dependency
would mix durable ownership with algorithm ownership and could constrain later
composition. Gate 1 must place shared vector identities and canonical numeric
types without creating a dependency cycle.

## Findings that constrain Gate 1

### 1. Disk format must be decided first

Format-1 log frames, mutation tags, snapshots, and restore logic are closed
over the current KV model. Adding new tags while retaining the format marker
could make an old binary begin replay and fail only after reading new state.
Gate 1 must choose an explicit version and migration rule before any vector
bytes are written.

### 2. Snapshot evolution is mandatory

Backup contains the logical snapshot, not the append log. If vectors are
missing from the snapshot, backup/restore cannot preserve them. Snapshot
format and disk-format evolution must be designed together.

### 3. Numeric scoring is a public compatibility decision

Current exact retrieval ranks `f64` cosine scores. Proof bytes, JSON values,
cross-platform ties, and future hybrid fusion require one canonical score
representation. Gate 1 must benchmark and specify the choice before proof or
HTTP implementation.

### 4. Retrieval proof should not mutate result-proof-v1 in place

`result-proof-v1` has existing magic, version, and operation semantics. A
separate proof format is easier to evolve and preserves existing verifiers.
The final choice remains subject to ADR-0016.

### 5. Shared type ownership is unresolved

Vector identity is needed by storage, retrieval, engine, contracts, and proof
code. Placing it in the wrong crate can create a dependency cycle or make a
public algorithm type authoritative for disk encoding. ADR-0013 must define
ownership before code moves.

### 6. Contract mirroring needs a checked workflow

Canonical contracts and packaged crate assets are intentionally duplicated
and checked for byte equality. The first contract change should add or
document a deterministic synchronization command rather than relying on
manual copying.

## Risk register

| Risk | Consequence | Required control |
|---|---|---|
| Reusing disk format `1` for new logical records | Old binaries may partially replay new state | Explicit version/migration ADR and fixture |
| Storing vectors only in Redb | Backup, restore, and rebuild lose authority | Log and logical snapshot ownership |
| Platform-dependent floating ordering | Different ranks and invalid proofs | Canonical score spec and golden vectors |
| Partial vector batches | Object/vector state divergence | Atomic mutation and failure tests |
| Extending `result-proof-v1` in place | Existing verifier incompatibility | Separate versioned retrieval proof unless disproven |
| Storage depending on retrieval algorithms | Dependency and ownership coupling | Shared domain boundary ADR |
| Unversioned Unicode/tokenization | Lexical results drift by platform/library | Pinned normalization semantics and fixtures |
| Implicit hybrid fallback | Unverifiable behavior change | Typed modality absence and explicit fusion version |
| Contract copies drift | Published crate differs from repository API | Deterministic sync plus equality tests |

## Gate decision

Gate 0 is complete for local planning. No 0.2 implementation was added.

The next change is Gate 1: ADR-0013 through ADR-0018, normative specifications,
golden vectors, and contract sketches. Disk-format evolution is the first
decision because every durable vector task depends on it.
